<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tessera app -> Vault | Credentials cross network boundary via AppRole auth; Vault is external trust domain |
| Actuator health endpoint -> caller | Health details (Vault version, seal state) could leak infrastructure info |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Information Disclosure | /actuator/health vault details | mitigate | `management.endpoint.health.show-details=when-authorized` already configured in application.yml — unauthenticated callers see only UP/DOWN status, not Vault version or seal state |
| T-09-02 | Information Disclosure | VAULT_ROLE_ID / VAULT_SECRET_ID in env | mitigate | AppRole credentials injected via Docker Compose environment at deploy time, never committed to git; .env.example documents the variable names without values |
| T-09-03 | Denial of Service | Vault unavailable at startup (prod) | accept | Prod profile uses `spring.config.import=vault://` (no `optional:` prefix) — fail-fast is correct behavior; app should not start with missing secrets. Dev/test uses `spring.cloud.vault.enabled=false` to avoid dependency on Vault |
| T-09-04 | Spoofing | Vault TLS verification | mitigate | Spring Cloud Vault defaults to verifying Vault server TLS certificate; `spring.cloud.vault.ssl.trust-store` can be configured for custom CA in production |
</threat_model>
