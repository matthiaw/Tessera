<objective>
Create the SchemaChangeEvent application event in fabric-core and wire it end-to-end: SchemaRegistry publishes it after every schema mutation, SqlViewProjection listens and regenerates views, SpringAiMcpAdapter listens and notifies MCP clients.

Purpose: Close the integration gap where schema mutations in SchemaRegistry never notify projections, causing SQL views to go stale and MCP clients to serve outdated tool lists until restart.

Output: SchemaChangeEvent record, SchemaRegistry publishing from all 8 mutating methods, two @EventListener methods, unit tests for publishing, unit tests for listener wiring, integration test proving SQL view regeneration on schema change.
</objective>

## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | SchemaChangeEvent payload | accept | Event carries only (modelId, changeType, typeSlug) — all three are already visible to any code with SchemaRegistry access. No PII, no credentials, no cross-tenant data. In-process only, never serialized to network or log at DEBUG+ level. |
| T-07-02 | Tampering | @EventListener bypasses auth | accept | Listeners run in the same JVM thread as the SchemaRegistry @Transactional method — the caller has already been authenticated/authorized at the REST/MCP layer before reaching SchemaRegistry. No new authorization surface. |
| T-07-03 | Denial of Service | regenerateForTenant called per schema change | accept | Schema changes are human-driven API calls (low frequency). regenerateForTenant executes a handful of CREATE OR REPLACE VIEW DDL statements per tenant — sub-second. No amplification risk at MVP scale. |
</threat_model>
