# Phase 02.5 Plan 03: Entity Resolution & Review Queue Summary

**Three-tier entity resolution (exact, embedding, fuzzy) in fabric-rules with EmbeddingService for pgvector similarity queries, plus extraction review queue REST API with accept/reject/override endpoints flowing through GraphService.apply()**

## Accomplishments
- EntityResolutionService implements three-tier deterministic resolution: exact name+type match, pgvector embedding cosine similarity (configurable threshold), Levenshtein fuzzy name matching
- EmbeddingService wraps Spring AI EmbeddingModel with pgvector store (ON CONFLICT upsert) and similarity query (<=> cosine distance operator)
- FuzzyNameMatcher uses Apache Commons Text LevenshteinDistance with normalized similarity score
- ResolutionResult sealed interface with Match (UUID + tier + score), Create, ReviewQueue outcomes
- ExtractionReviewController at /admin/extraction/review: GET list (with filters), POST accept, POST reject, POST override
- Accepted/overridden candidates flow through GraphService.apply() with confidence 1.0 (operator-affirmed)
- All endpoints enforce tenant isolation via JWT model_id; cross-tenant access returns 404
- 12 unit tests (EntityResolutionServiceTest + FuzzyNameMatcherTest) and 5 integration tests (ExtractionReviewControllerIT)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Circular dependency avoidance**
- **Found during:** Task 1
- **Issue:** Plan specifies EntityResolutionService.resolve() accepts ExtractionCandidate, but ExtractionCandidate lives in fabric-connectors. Adding fabric-connectors as a dependency of fabric-rules would create a circular dependency.
- **Fix:** Created ResolutionCandidate record in fabric-rules as a lightweight input type. Connectors layer maps ExtractionCandidate to ResolutionCandidate before calling resolution.
- **Files modified:** ResolutionCandidate.java (new)
- **Committed in:** a30a44d (Task 1 commit)

**2. [Rule 3 - Blocking] Spring AI auto-config bean conflicts in IT**
- **Found during:** Task 2
- **Issue:** Spring AI auto-configuration creates both anthropicChatModel and ollamaChatModel beans, causing NoUniqueBeanDefinitionException. ExtractionService also requires SchemaRegistry which isn't available in the test context.
- **Fix:** Excluded Spring AI auto-configurations by name in @EnableAutoConfiguration, provided mock beans for ChatModel, EmbeddingModel, and SchemaRegistry
- **Files modified:** ExtractionReviewControllerIT.java
- **Committed in:** ad46995 (Task 2 commit)

**3. [Rule 3 - Blocking] JwtTestUtil package-private access**
- **Found during:** Task 2
- **Issue:** JwtTestUtil and its methods were package-private in dev.tessera.connectors.rest, inaccessible from dev.tessera.connectors.review package
- **Fix:** Made class and mintAdmin/mint methods public
- **Files modified:** JwtTestUtil.java
- **Committed in:** ad46995 (Task 2 commit)
