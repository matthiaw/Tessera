## Cross-Phase Integration Gaps

### 1. SchemaRegistry → SqlViewProjection / SpringAiMcpAdapter (MCP-08, SQL-02)

`SchemaRegistry.createNodeType()` persists to DB but publishes no `ApplicationEvent`. `SqlViewProjection` regenerates views only at startup via `ApplicationRunner` — no `@EventListener` for runtime schema changes. `SpringAiMcpAdapter.notifySchemaChanged()` exists but is never called — no `SchemaChangeEvent` class in fabric-core. Phase 7 was designed to close this gap but was not executed.

### 2. CircleadConnectorConfig → GenericRestPollerConnector (CIRC-01, CIRC-02)

`CircleadConnectorConfig` loads 3 `MappingDefinition` JSON files whose `sourceUrl` fields contain `${tessera.connectors.circlead.base-url}/...`. Jackson `ObjectMapper.readValue()` does not resolve Spring property placeholders. `GenericRestPollerConnector.poll()` passes the literal `${...}` string to `URI.create()` which throws `IllegalArgumentException`. Phase 8 was designed to wire ConnectorInstance objects but was not executed.

### 3. OutboxPoller → downstream consumers (EVENT-05 — by design)

`OutboxPoller` publishes `GraphEventPublished` via `ApplicationEventPublisher` but no production `@EventListener` exists. In-process event bus delivers nothing to downstream consumers. This is by design — Kafka/Debezium (Phase 4) is the intended consumer path, and `tessera.kafka.enabled=false` by default conditionalizes the OutboxPoller.
