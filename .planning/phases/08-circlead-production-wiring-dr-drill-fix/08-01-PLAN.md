<objective>
Fix the two circlead connector wiring bugs: (1) resolve Spring `${...}` placeholders in `sourceUrl` before `URI.create()` sees them, and (2) register circlead connector instances in the `connectors` DB table so `ConnectorRegistry.loadAll()` picks them up and `ConnectorScheduler.tick()` dispatches syncs.

Purpose: Without these fixes, circlead connectors exist only as Spring beans but are invisible to the scheduler. `GenericRestPollerConnector.poll()` throws `IllegalArgumentException` from `URI.create()` because the placeholder is never resolved.

Output: Working `CircleadConnectorConfig` with placeholder resolution + DB upsert, application.yml config key, unit test proving resolution.
</objective>

       return new MappingDefinition(
           raw.sourceEntityType(), raw.targetNodeTypeSlug(), raw.rootPath(),
           raw.fields(), raw.identityFields(), resolvedUrl,
           raw.folderPath(), raw.globPattern(), raw.chunkStrategy(),
           raw.chunkOverlapChars(), raw.confidenceThreshold(), raw.provider());
   }

5. **Add `@PostConstruct` method** to upsert connector rows into the `connectors` table:
   ```java
   @PostConstruct
   void registerCircleadConnectors() {
       try {
           List<MappingDefinition> mappings = List.of(
               circleadRoleMapping(), circleadCircleMapping(), circleadActivityMapping());
           for (MappingDefinition m : mappings) {
               jdbc.update("""
                   INSERT INTO connectors (model_id, type, mapping_def, auth_type, credentials_ref,
                                           poll_interval_seconds, enabled)
                   VALUES (:modelId::uuid, 'rest-poll', :mappingJson::jsonb, 'BEARER',
                           :credentialsRef, :interval, true)
                   ON CONFLICT DO NOTHING
                   """,
                   Map.of("modelId", circleadModelId,
                          "mappingJson", objectMapper.writeValueAsString(m),
                          "credentialsRef", circleadCredentialsRef,
                          "interval", circleadPollIntervalSeconds));
           }
           LOG.info("CircleadConnectorConfig: registered {} circlead connector(s)", mappings.size());
       } catch (Exception e) {
           LOG.warn("CircleadConnectorConfig: failed to register circlead connectors (table may not exist yet): {}",
               e.getMessage());
       }
   }
   ```
   Add `private static final Logger LOG = LoggerFactory.getLogger(CircleadConnectorConfig.class);` and import `org.slf4j.Logger` + `org.slf4j.LoggerFactory`.

**Important:** The `@PostConstruct registerCircleadConnectors()` must run BEFORE `ConnectorRegistry.loadAll()`. Since both are `@PostConstruct` and `CircleadConnectorConfig` is in `fabric-connectors` alongside `ConnectorRegistry`, use `@DependsOn` or `@Order` if needed. However, Spring typically processes `@Configuration` beans before `@Component` beans, so `CircleadConnectorConfig` (a `@Configuration`) will initialize before `ConnectorRegistry` (a `@Component`). If this ordering is not reliable, annotate `ConnectorRegistry` with `@DependsOn("circleadConnectorConfig")` or have `ConnectorRegistry` constructor-inject `CircleadConnectorConfig` to force dependency ordering.

Alternatively, since `ConnectorRegistry.loadAll()` catches exceptions gracefully and logs warnings, a simpler approach: add a `@EventListener(ApplicationReadyEvent.class)` to `ConnectorRegistry` that calls `loadAll()` again, or make the upsert happen via an `ApplicationRunner` bean. The cleanest path: keep `@PostConstruct` on `CircleadConnectorConfig` and change `ConnectorRegistry.loadAll()` from `@PostConstruct` to `@EventListener(ContextRefreshedEvent.class)` — but this changes existing behavior. Evaluate and pick the least invasive approach. The `ON CONFLICT DO NOTHING` makes repeated registration safe regardless.
