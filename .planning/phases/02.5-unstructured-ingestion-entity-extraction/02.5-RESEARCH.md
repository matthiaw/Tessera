# Phase 2.5: Unstructured Ingestion & Entity Extraction - Research

**Researched:** 2026-04-16
**Domain:** LLM-based entity extraction, pgvector similarity search, Spring AI structured output, entity resolution
**Confidence:** HIGH

## Summary

Phase 2.5 extends Tessera's connector framework with LLM-based entity extraction from unstructured text. The technical surface spans four domains: (1) Spring AI 1.0.5 structured output with Anthropic Claude for extraction, (2) pgvector for embedding-based entity resolution, (3) Ollama for self-hosted embedding generation, and (4) text chunking + fuzzy matching for entity deduplication. All four domains are well-served by mature libraries already in or compatible with Tessera's stack.

The critical finding is that the `apache/age` Docker image does NOT include pgvector -- a custom Dockerfile extending the AGE image with pgvector compiled from source is required. This affects both the Testcontainers test harness and the production Docker Compose stack. The custom image pattern is well-documented in the community and straightforward to implement.

The existing codebase is well-prepared for Phase 2.5: `GraphMutation` already carries `extractorVersion` and `llmModelId` fields (forward-committed in Phase 1), `SourceType.EXTRACTION` already exists in the enum, and the Spring AI BOM 1.0.5 is already declared in the parent POM. The main additions are: two new fields on `GraphMutation` (`sourceDocumentId`, `sourceChunkRange`), corresponding columns on `graph_events`, the pgvector extension + tables via Flyway, and the `connectors` table `auth_type` CHECK constraint needs widening to accept `NONE` (folder connectors have no credentials).

**Primary recommendation:** Build a custom `tessera/age-pgvector:PG16` Docker image extending `apache/age` with pgvector compiled from source. Use this for both Docker Compose and Testcontainers. All other components (Spring AI, Commons Text, Ollama) are drop-in additions to the existing stack.

### Pattern 1: Extraction Pipeline Flow

**What:** The full data flow from file scan to graph mutation.
**When to use:** Every unstructured connector poll cycle.

```
MarkdownFolderConnector.poll()
  → scan folder, compute SHA-256, skip unchanged files
  → for each changed file:
      → TextChunker.chunk(fileContent, config)
      → for each chunk:
          → ExtractionService.extract(chunk, tenantContext)
              → SchemaRegistrySchemaBuilder.buildExtractionSchema(tenantContext)
              → DynamicSchemaOutputConverter (format instructions)
              → ChatClient.prompt().user(...).call().content()
              → converter.convert(response) → List<ExtractionCandidate>
          → for each ExtractionCandidate:
              → build CandidateMutation with provenance fields
  → return PollResult(candidates, nextState, outcome, dlq)

ConnectorRunner.runOnce(instance)
  → for each CandidateMutation:
      → EntityResolutionService.resolve(candidate, tenantContext)
          → Tier 1: exact name+type match in AGE
          → Tier 2: pgvector cosine similarity ≥ threshold
          → Tier 3: Levenshtein fuzzy match
          → Result: MATCH(uuid) | CREATE | REVIEW_QUEUE
      → if REVIEW_QUEUE: write to extraction_review_queue, skip apply
      → if MATCH: set targetNodeUuid on candidate
      → candidate.toMutation(tenant) → GraphMutation
      → graphService.apply(mutation)
      → EmbeddingService.storeEmbedding(nodeUuid, entityName, tenantContext) [post-apply]
```

### Pattern 2: ConnectorRunner Extension Point

**What:** Where entity resolution and embedding storage plug into the existing `ConnectorRunner`.
**When to use:** The runner needs modification to call `EntityResolutionService` before `graphService.apply()` and `EmbeddingService` after.

The current `ConnectorRunner.runOnce()` iterates `CandidateMutation` candidates and calls `candidate.toMutation(tenant)` then `graphService.apply(mutation)`. Phase 2.5 inserts two hooks:

1. **Pre-apply (entity resolution):** Between `candidate.toMutation()` and `graphService.apply()`, call `EntityResolutionService.resolve()`. This may:
   - Set `targetNodeUuid` on the candidate (merge path → UPDATE operation)
   - Leave `targetNodeUuid` null (create path)
   - Route to `extraction_review_queue` (skip apply entirely)

2. **Post-apply (embedding):** After `graphService.apply()` returns `Committed`, call `EmbeddingService.storeEmbedding()` to generate and persist the embedding for the newly created/updated node.

**Key constraint:** Entity resolution runs BEFORE apply. Embedding storage runs AFTER apply. The runner is the orchestrator; neither `EntityResolutionService` nor `EmbeddingService` may call `GraphService` directly (ArchUnit enforced). [VERIFIED: CONTEXT.md Decision 5]

### Pattern 3: CandidateMutation Provenance Extension

**What:** Adding provenance fields to `CandidateMutation` without breaking Phase 2 callers.
**When to use:** Phase 2.5 extraction path produces candidates with 5 provenance fields.

The current `CandidateMutation` record has 6 fields. Java records are final and cannot be subclassed. Options:

**Recommended approach:** Add the 5 provenance fields as new record components with `@Nullable` semantics. Update `toMutation()` to pass them through. Phase 2's `GenericRestPollerConnector` passes `null` for all 5 -- no behavior change.

```java
public record CandidateMutation(
        String targetTypeSlug,
        UUID targetNodeUuid,
        Map<String, Object> properties,
        String sourceSystem,
        String connectorId,
        String changeId,
        // Phase 2.5 provenance (null for structured connectors)
        String sourceDocumentId,
        String sourceChunkRange,
        String extractorVersion,
        String llmModelId,
        BigDecimal extractionConfidence) {
    // ...
}
```

The existing `GraphMutation` record already has `extractorVersion` and `llmModelId` but is missing `sourceDocumentId` and `sourceChunkRange`. These two fields must be added to `GraphMutation` as well, and `toMutation()` must thread them through. [VERIFIED: GraphMutation.java, CandidateMutation.java in codebase]

### Anti-Patterns to Avoid

- **Extraction service calling GraphService directly:** Only `ConnectorRunner` may call `GraphService.apply()`. ArchUnit enforces this. The extraction service produces `ExtractionCandidate` records, which `MarkdownFolderConnector` converts to `CandidateMutation`.
- **Embedding generation inside GraphService.apply() transaction:** Embedding calls hit Ollama over HTTP -- slow, may time out. Must happen post-apply, outside the graph transaction.
- **Blocking on Ollama in the poll thread:** Use virtual threads (`spring.threads.virtual.enabled=true`) so the HTTP call to Ollama does not block platform threads.
- **Hardcoding JSON schema in connector config:** The schema MUST come from the Schema Registry at runtime (Decision 2). No pre-baked extraction schemas.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Fuzzy string matching | Custom Levenshtein | `org.apache.commons.text.similarity.LevenshteinDistance` | Well-tested, handles edge cases (null, empty), configurable threshold |
| JSON schema generation from types | Manual schema string builder | Jackson `JsonSchemaGenerator` or manual but simple builder walking `NodeTypeDescriptor` | Schema must match Schema Registry exactly |
| Vector similarity queries | Custom distance math | pgvector `<=>` operator with `vector_cosine_ops` | Hardware-accelerated, index-backed, deterministic |
| Embedding API abstraction | Custom HTTP client to Ollama | Spring AI `EmbeddingModel` interface | Provider-agnostic, swap Ollama for OpenAI by changing starter dependency |
| LLM structured output parsing | Manual JSON extraction from free text | Spring AI `StructuredOutputConverter` pattern | Handles format instructions + parsing in one abstraction |
| Retry with backoff | Custom retry loop | Spring Retry `@Retryable` or manual loop with `Thread.sleep` (virtual threads safe) | Extraction calls can fail with malformed JSON -- retry is essential |

## Common Pitfalls

### Pitfall 1: apache/age Docker Image Lacks pgvector

**What goes wrong:** `CREATE EXTENSION IF NOT EXISTS vector;` fails with "extension not found" because the official `apache/age` image does not include pgvector.
**Why it happens:** The AGE image is based on vanilla Postgres with only the AGE extension compiled in.
**How to avoid:** Build a custom Dockerfile extending `apache/age` (or building from `postgres:16-bookworm`) with pgvector compiled from source. Use this image for both Docker Compose and Testcontainers.
**Warning signs:** Flyway V16 migration fails on container startup; `SELECT * FROM pg_available_extensions WHERE name = 'vector'` returns empty.

**Custom Dockerfile pattern:**
```dockerfile
FROM apache/age@sha256:16aa423d20a31aed36a3313244bf7aa00731325862f20ed584510e381f2feaed

RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      build-essential git postgresql-server-dev-16 && \
    cd /tmp && \
    git clone --branch v0.8.0 https://github.com/pgvector/pgvector.git && \
    cd pgvector && \
    make && make install && \
    cd / && rm -rf /tmp/pgvector && \
    apt-get remove -y build-essential git postgresql-server-dev-16 && \
    apt-get autoremove -y && \
    rm -rf /var/lib/apt/lists/*
```
[CITED: github.com/cromulus/postgres-with-age-pgvector Dockerfile pattern]

### Pitfall 2: Spring AI 1.0.x Default max_tokens Is 500

**What goes wrong:** Extraction response is truncated mid-JSON, causing `JsonProcessingException` on every call. Entity extraction requires 2000-4000 output tokens for a typical chunk.
**Why it happens:** Spring AI Anthropic starter defaults `max-tokens` to 500 if not explicitly set.
**How to avoid:** Always set `AnthropicChatOptions.builder().maxTokens(4096)` on every extraction call, or configure `spring.ai.anthropic.chat.options.max-tokens=4096` globally.
**Warning signs:** Extraction calls consistently fail with JSON parse errors; truncated response text visible in logs.
[VERIFIED: Spring AI Anthropic docs]

### Pitfall 3: BeanOutputConverter Cannot Handle Runtime Schemas

**What goes wrong:** Developer tries to use `BeanOutputConverter<ExtractionCandidate>` with a compile-time Java class, but the extraction schema must vary per tenant based on Schema Registry state.
**Why it happens:** `BeanOutputConverter` generates a JSON schema from a fixed Java class at construction time.
**How to avoid:** Implement a custom `StructuredOutputConverter<List<ExtractionCandidate>>` that takes the JSON schema string as a constructor parameter. The `getFormat()` method returns format instructions derived from the runtime schema. The `convert()` method parses the LLM response JSON.
**Warning signs:** All tenants get the same extraction schema regardless of their Schema Registry configuration.
[VERIFIED: Spring AI StructuredOutputConverter docs]

### Pitfall 4: connectors Table auth_type CHECK Constraint

**What goes wrong:** `INSERT INTO connectors (..., auth_type, ...) VALUES (..., 'NONE', ...)` fails because V14 migration has `CHECK (auth_type IN ('BEARER'))`.
**Why it happens:** Phase 2 only supported Bearer-authenticated REST connectors. Folder connectors have no credentials -- they need `auth_type = 'NONE'` and `credentials_ref` should be nullable.
**How to avoid:** Phase 2.5 Flyway migration must `ALTER TABLE connectors DROP CONSTRAINT ...` and replace with `CHECK (auth_type IN ('BEARER', 'NONE'))`. Also `ALTER TABLE connectors ALTER COLUMN credentials_ref DROP NOT NULL`.
**Warning signs:** Connector creation for `type = "unstructured-text"` fails with constraint violation.
[VERIFIED: V14__connectors.sql in codebase]

### Pitfall 5: pgvector and AGE Extension Load Order

**What goes wrong:** `LOAD 'age'` and `CREATE EXTENSION vector` interact unexpectedly if the shared library load order is wrong.
**Why it happens:** Both extensions install shared libraries. AGE requires explicit `LOAD 'age'` in the session init.
**How to avoid:** Install pgvector via `CREATE EXTENSION IF NOT EXISTS vector;` in a Flyway migration AFTER the AGE baseline (V1). The `LOAD 'age'` happens at connection init via HikariCP `connectionInitSql` -- this is independent of pgvector. Both extensions coexist without conflict because they operate on different data types and operators.
**Warning signs:** `LOAD 'age'` fails after pgvector install, or `vector` type is not recognized.
[CITED: medium.com/percolation-labs/cloudnativepg-age-and-pg-vector article confirms coexistence]

### Pitfall 6: Embedding Dimension Mismatch

**What goes wrong:** Entity resolution cosine similarity queries return wrong results or errors because embeddings were stored with one dimension (768 for Ollama) but queried with another (1536 for OpenAI).
**Why it happens:** Different embedding models produce different vector dimensions.
**How to avoid:** The `entity_embeddings` table records `embedding_model TEXT` per row. The `EntityResolutionService` must only compare embeddings from the same model. The HNSW index must be created with a fixed dimension that matches the default model (768). If a tenant uses OpenAI (1536-dim), they need a separate index or must normalize to 768.
**Warning signs:** Cosine similarity always returns 0 or throws dimension mismatch errors.
[ASSUMED -- standard pgvector behavior]

## Code Examples

### Example 1: DynamicSchemaOutputConverter

```java
// Custom StructuredOutputConverter for runtime Schema Registry schemas
// Source: Spring AI StructuredOutputConverter interface pattern [VERIFIED: Spring AI docs]
public class DynamicSchemaOutputConverter
        implements StructuredOutputConverter<List<ExtractionCandidate>> {

    private final String jsonSchema;
    private final ObjectMapper objectMapper;

    public DynamicSchemaOutputConverter(String jsonSchema) {
        this.jsonSchema = jsonSchema;
        this.objectMapper = new ObjectMapper();
    }

    @Override
    public String getFormat() {
        return """
            Respond with a JSON array of extracted entities matching this schema:
            %s
            Each entity MUST include:
            - "typeSlug": the entity type slug from the schema
            - "name": the entity's primary name as it appears in the text
            - "properties": object matching the type's declared properties
            - "sourceSpan": {"charOffset": <int>, "charLength": <int>}
            - "confidence": a number between 0.0 and 1.0
            - "relationships": array of {"edgeType": "<slug>", "targetName": "<name>", "targetType": "<slug>"}
            If no entities found, return an empty array [].
            Do NOT invent entities not present in the text.
            """.formatted(jsonSchema);
    }

    @Override
    public List<ExtractionCandidate> convert(String text) {
        String cleaned = text.strip();
        // Strip markdown code fences if present
        if (cleaned.startsWith("```")) {
            cleaned = cleaned.replaceAll("^```json?\\n?", "")
                             .replaceAll("```$", "")
                             .strip();
        }
        return objectMapper.readValue(cleaned,
                new TypeReference<List<ExtractionCandidate>>() {});
    }
}
```

### Example 2: pgvector Cosine Similarity Query

```sql
-- Entity resolution tier 2: find nearest existing entity by embedding
-- Source: pgvector README [VERIFIED: github.com/pgvector/pgvector]
SELECT e.node_uuid, 1 - (e.embedding <=> :query_embedding) AS cosine_similarity
FROM entity_embeddings e
WHERE e.model_id = :model_id
  AND e.embedding_model = :embedding_model
ORDER BY e.embedding <=> :query_embedding
LIMIT 5;
```

In Java via `JdbcTemplate`:
```java
// Note: pgvector requires casting the parameter to ::vector
String sql = """
    SELECT node_uuid, 1 - (embedding <=> ?::vector) AS similarity
    FROM entity_embeddings
    WHERE model_id = ? AND embedding_model = ?
    ORDER BY embedding <=> ?::vector
    LIMIT 5
    """;
```

## Open Questions

1. **Exact pgvector version for PG16**
   - What we know: pgvector 0.7.x and 0.8.x support PG16. Community images use various versions.
   - What's unclear: Whether `v0.8.0` tag exists or needs `v0.7.4`. Minor detail.
   - Recommendation: Use `git clone --branch v0.8.0` in Dockerfile; if tag doesn't exist, fall back to latest release tag. Build will fail fast if incompatible.

2. **AGE Docker image base OS**
   - What we know: Image is `apache/age` pinned by digest. Community forks use `postgres:16-bookworm`.
   - What's unclear: Whether the specific pinned digest is Debian-based (allowing `apt-get`) or otherwise.
   - Recommendation: Run `docker run --rm apache/age@sha256:16aa... cat /etc/os-release` to confirm before writing the Dockerfile.

3. **Spring AI 1.0.5 Anthropic native structured output availability**
   - What we know: Spring AI docs list Anthropic Claude 3.5 Sonnet+ as supporting native structured output via `AdvisorParams.ENABLE_NATIVE_STRUCTURED_OUTPUT`. AI-SPEC says 1.0.x is prompt-based only.
   - What's unclear: Whether 1.0.5 actually supports native structured output for Anthropic, or if this was added in 1.1.x.
   - Recommendation: Use prompt-based structured output (custom `DynamicSchemaOutputConverter`) regardless. It works, and SHACL provides the second validation layer. Try native output as a future optimization.

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| LLM hallucinated entities | Tampering | Source span verification, confidence threshold, review queue |
| Prompt injection via document content | Tampering | System/user prompt separation; extracted text in user prompt only, not system prompt |
| API key leakage in logs | Information Disclosure | Vault-loaded key; never log API key; Spring AI does not log keys by default |
| Review queue IDOR | Elevation of Privilege | Tenant-scoped queries (`WHERE model_id = ?`), JWT tenant claim enforcement |
| Denial of service via large documents | Denial of Service | Chunk size limits, max entity count guardrail (50 per chunk), extraction rate limit |
