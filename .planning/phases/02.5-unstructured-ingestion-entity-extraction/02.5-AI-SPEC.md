# AI-SPEC — Phase 2.5: Unstructured Ingestion & Entity Extraction

> AI design contract generated by `/gsd-ai-integration-phase`. Consumed by `gsd-planner` and `gsd-eval-auditor`.
> Locks framework selection, implementation guidance, and evaluation strategy before planning begins.

---

## 1. System Classification

**System Type:** Extraction

**Description:**
LLM-based entity and relationship extractor that processes unstructured text (markdown files, wiki pages, chat logs) and produces typed graph entities conforming to a runtime Schema Registry. The system sits inside a connector pipeline — extracted candidates flow through the same rule engine, SHACL validation, reconciliation, and source authority matrix as structured connectors. Operators configure extraction via connector admin endpoints; low-confidence extractions land in a review queue for human decision.

**Critical Failure Modes:**
1. **Hallucinated entities** — LLM invents entities not present in source text, corrupting the knowledge graph silently
2. **False entity merges** — Entity resolution incorrectly merges two distinct real-world entities into one graph node, destroying data integrity
3. **Provenance loss** — Extracted entities lack traceable lineage to source document/chunk, making the graph unauditable
4. **Schema violation** — LLM output doesn't conform to Schema Registry types, causing silent data loss or SHACL rejection storms
5. **Confidence miscalibration** — Review queue threshold is meaningless if LLM confidence scores don't correlate with actual accuracy

## 1b. Domain Context

**Industry Vertical:** Enterprise data integration / knowledge graph construction

**User Population:** System operators configuring unstructured connectors and reviewing low-confidence extractions; downstream LLM agents and BI tools querying the graph

**Stakes Level:** High — the graph is the single source of truth; bad extraction data propagates to all projections (REST, MCP, SQL views, Kafka)

**Output Consequence:** Extracted entities become first-class graph nodes indistinguishable from structured-connector data. False data in the graph leads to wrong answers from LLM agents, incorrect BI reports, and broken integration flows.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes |
|-----------|----------------------|-------------------|--------|
| Entity detection | All named entities in source text appear as graph nodes with correct types | Missing entities, or entities invented by the LLM that don't appear in source | High — missed entities = incomplete graph; hallucinated = corrupted graph |
| Type classification | "Acme Corp" → Organization, "Jane Smith" → Person | "Acme Corp" → Person, or novel types not in Schema Registry | Medium — wrong type breaks SHACL validation and downstream queries |
| Relationship extraction | "Jane works at Acme" → WORKS_AT edge with correct direction | Missing relationships, wrong direction, or hallucinated relationships | High — relationships are the graph's primary value |
| Property completeness | Email, role, department extracted when present in text | Properties present in text but not extracted; or properties hallucinated | Medium — missing is annoying; hallucinated is dangerous |
| Entity resolution | "IBM" and "International Business Machines" merge into one node | Two distinct "John Smith" people merged into one; or same person creates duplicates | Critical — false merge corrupts; duplicate is fixable |

### Known Failure Modes in This Domain

1. **Silent hallucination** — LLM generates plausible-sounding entities that pass schema validation but don't exist in source text. Undetectable without source span verification.
2. **Input sensitivity** — Removing a single sentence or changing punctuation can cause dramatic output changes (GDELT Project documented: one apostrophe change triggered hallucination, 5-word removal collapsed output from 1024 to 183 tokens).
3. **Over-extraction** — LLM splits one real entity into multiple graph nodes (e.g., "Dr. Smith" and "Jane Smith" as separate Person nodes from the same document).
4. **Confidence overconfidence** — LLMs are systematically overconfident. ECE of 0.05–0.20 is typical. Raw confidence of 0.8 does NOT mean 80% accuracy.
5. **Format degradation** — Under certain prompts or long inputs, LLM stops producing valid JSON mid-output, causing silent data loss for entities after the parse failure point.

### Regulatory / Compliance Context

No direct regulatory constraints on extraction accuracy. However, Tessera's design supports optional hash-chained audit (Phase 4) and GxP/SOX/BSI C5 compliance. Extraction provenance (EXTR-04) is a prerequisite for audit trail integrity — every extracted datum must trace back to its source document and chunk.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Operator (connector admin) | Review queue decisions (accept/reject/override); calibration dataset labeling; extraction quality monitoring |
| Graph consumer (LLM agent / BI analyst) | Reports incorrect data found via queries — signals extraction quality regressions |

## 2. Framework Decision

**Selected Framework:** Spring AI 1.0.5 (stable 1.0.x line)

**Version:** 1.0.5 — pin strictly. Do not upgrade to 1.1.x or 2.0.0-Mx without testing.

**Rationale:**
Spring AI 1.0.x integrates natively with Spring Boot 3.5.x (Tessera's stack). The `spring-ai-anthropic-spring-boot-starter` provides Claude chat integration with prompt-based structured output via `BeanOutputConverter`. Native Anthropic structured output (JSON schema sent to API) landed in 2.0.0-M1 only — not backported to 1.0.x. Prompt-based structured output is sufficient because Tessera validates extracted entities against SHACL shapes post-extraction, providing a second validation layer.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain4j | Java-native alternative, but Spring AI is already in the project roadmap (Phase 3 MCP). Two LLM frameworks creates confusion. Documented as fallback in EXTR-03. |
| Direct Anthropic Java SDK (`io.modelcontextprotocol:sdk-java`) | More control but loses Spring Boot auto-configuration, ChatClient fluent API, and EmbeddingModel abstraction. |
| Python-based frameworks (LangChain, LlamaIndex, CrewAI) | Tessera is Java 21 — adding Python creates a second runtime, build system, and deployment artifact. |

**Vendor Lock-In Accepted:** Partial — Anthropic for chat/extraction (locked in CONTEXT.md), but embeddings are provider-configurable via Spring AI's `EmbeddingModel` interface. Switching extraction provider requires only changing the Spring AI starter dependency and Vault secret path.

### Entry Point Pattern

```java
// Extraction call with dynamic schema from Schema Registry
ChatClient chatClient = ChatClient.create(chatModel);

String schemaJson = schemaRegistrySchemaBuilder.buildExtractionSchema(tenantContext);
var converter = new DynamicSchemaOutputConverter(schemaJson);

Map<String, Object> result = chatClient.prompt()
    .user(u -> u.text("""
        Extract all entities and relationships from the following text.
        Return ONLY valid JSON matching the schema below.
        For each entity, include the exact character offset where it appears.
        
        Text: {text}
        
        {format}
        """)
        .param("text", chunk.text())
        .param("format", converter.getFormat()))
    .options(AnthropicChatOptions.builder()
        .model("claude-sonnet-4-5")
        .maxTokens(4096)
        .temperature(0.0)
        .build())
    .call()
    .content();

List<ExtractionCandidate> candidates = converter.convert(result);
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `ChatClient` | Fluent API wrapping `ChatModel` (like `RestClient` wraps `RestTemplate`) | Every extraction call — provides `.prompt().user().call()` chain |
| `ChatModel` | Low-level model abstraction; one impl per provider | Injected as Spring bean; mocked in unit tests |
| `StructuredOutputConverter<T>` | Interface with `getFormat()` (schema instructions) and `convert()` (parse response) | Custom `DynamicSchemaOutputConverter` for runtime schema from Schema Registry |
| `EmbeddingModel` | Provider-agnostic embedding interface | Entity resolution similarity search — abstracts Ollama vs OpenAI |
| `AnthropicChatOptions` | Per-call options (model, maxTokens, temperature) | Override defaults per extraction call |

### Common Pitfalls

1. **Default `max_tokens` is 500** — far too low for entity extraction. Always set explicitly to 4096+.
2. **No built-in retry on malformed output** — `converter.convert()` throws `JsonProcessingException` on invalid JSON. You must implement retry with exponential backoff.
3. **`BeanOutputConverter` requires compile-time classes** — cannot accept dynamic JSON schemas. Use a custom `StructuredOutputConverter` implementation for runtime schemas from the Schema Registry.
4. **Spring AI 1.0.x Anthropic structured output is prompt-based only** — the LLM is not constrained to produce valid JSON. Temperature 0.0 + clear schema instructions mitigate but don't guarantee.
5. **Anthropic has no embedding API** — Spring AI's Anthropic starter does NOT include embeddings. Use a separate embedding provider (Ollama or OpenAI).

## 4. Implementation Guidance

**Model Configuration:**
- Model: `claude-sonnet-4-5` (best cost/quality for extraction; upgrade to Opus only if accuracy insufficient)
- Temperature: `0.0` (deterministic extraction — reproducibility is a requirement per EXTR-05)
- Max tokens: `4096` (sufficient for 20–30 entities per chunk; increase if chunks are very large)
- System prompt: Extraction instructions with schema + examples (few-shot)

**Core Pattern:**
Chunk text → build schema from Schema Registry → call Claude via `ChatClient` with `DynamicSchemaOutputConverter` → validate JSON output → verify source spans → compute confidence → route to entity resolution → either auto-merge via `GraphService.apply()` or queue for review.

**Tool Use:** Not applicable — extraction uses structured output, not tool calling.

**State Management:**
- `ConnectorState` JSONB: `file_hashes` map for dedup across polls
- `extraction_review_queue` table: pending decisions
- `entity_embeddings` table: pgvector storage for resolution

**Context Window Strategy:**
- Paragraph-based chunking keeps chunks well within Claude's context window (200k tokens)
- Chunk overlap (200 chars) preserves entity context across boundaries
- Schema instructions + chunk text + format instructions must fit within context; monitor with response metadata

## 4b. AI Systems Best Practices

### Structured Outputs — Custom Converter Pattern (Java)

Spring AI 1.0.x `BeanOutputConverter` requires compile-time types. For Tessera's dynamic Schema Registry, implement `StructuredOutputConverter`:

```java
public class DynamicSchemaOutputConverter implements StructuredOutputConverter<List<ExtractionCandidate>> {
    private final String jsonSchema;
    private final ObjectMapper objectMapper;

    public DynamicSchemaOutputConverter(String jsonSchema) {
        this.jsonSchema = jsonSchema;
        this.objectMapper = new ObjectMapper();
    }

    @Override
    public String getFormat() {
        return """
            Respond with a JSON array of extracted entities matching this schema:
            %s
            Each entity MUST include sourceSpan (character offset and length in source text).
            Each entity MUST include confidence (0.0-1.0).
            If no entities found, return an empty array [].
            """.formatted(jsonSchema);
    }

    @Override
    public List<ExtractionCandidate> convert(String text) {
        String cleaned = text.strip();
        if (cleaned.startsWith("```")) {
            cleaned = cleaned.replaceAll("^```json?\\n?", "").replaceAll("```$", "").strip();
        }
        return objectMapper.readValue(cleaned, new TypeReference<>() {});
    }
}
```

### Retry on Malformed Output

```java
public List<ExtractionCandidate> extractWithRetry(String chunk, String schema, int maxRetries) {
    for (int attempt = 0; attempt < maxRetries; attempt++) {
        try {
            String response = callLlm(chunk, schema);
            return converter.convert(response);
        } catch (JsonProcessingException e) {
            if (attempt == maxRetries - 1) throw new ExtractionException("Failed after " + maxRetries + " attempts", e);
            // Exponential backoff: 1s, 2s, 4s
        }
    }
}
```

### Virtual Threads for Extraction

Tessera uses Java 21 virtual threads (`spring.threads.virtual.enabled=true`). The extraction HTTP call to Anthropic is blocking I/O — virtual threads handle this naturally without WebFlux. No special async handling needed.

### Prompt Engineering Discipline

- **System prompt**: Extraction instructions, entity type definitions, few-shot examples (2–3 per entity type)
- **User prompt**: The text chunk + format instructions from `DynamicSchemaOutputConverter.getFormat()`
- Separate system and user prompts — system carries stable instructions, user carries per-chunk content
- Include negative examples: "Do NOT extract entities that are not explicitly mentioned in the text"

### Cost and Latency Budget

- Estimated cost per chunk: ~$0.003 (Sonnet, 2k input + 1k output tokens)
- Estimated cost per document (5 chunks avg): ~$0.015
- Estimated cost for 1000-document Obsidian vault: ~$15
- Embedding cost (Ollama): $0 (self-hosted)
- Embedding cost (OpenAI): ~$0.00002 per entity
- Latency per chunk: 2–5 seconds (Anthropic API)
- Latency per document: 10–25 seconds (sequential chunks; parallelizable with virtual threads)

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail) | Measurement Approach | Priority |
|-----------|-------------------|---------------------|----------|
| Entity detection F1 | F1 ≥ 0.70 on gold corpus (strict match) | Code — compare extracted vs. annotated entities | Critical |
| Type classification accuracy | ≥ 90% of detected entities have correct type | Code — per-type accuracy from gold corpus | Critical |
| Relationship extraction F1 | F1 ≥ 0.60 on gold corpus | Code — compare extracted vs. annotated relationships | High |
| Source span validity | 100% of extracted entities have valid spans in source text | Code — span verification against source document | Critical |
| Confidence calibration | ECE < 0.15 on calibration dataset | Code — binned calibration analysis | High |
| Entity resolution precision | False merge rate < 5% on resolution test pairs | Code — labeled match/non-match pairs | Critical |
| Entity resolution recall | Merge recall ≥ 70% on resolution test pairs | Code — labeled match/non-match pairs | High |
| Provenance completeness | 100% of extracted entities carry all 5 provenance fields | Code — assert non-null on event log entries | Critical |

### Reference Dataset

**Size:** 50 annotated documents to start; grow to 200+ over time

**Composition:**
- 30 markdown notes (Obsidian vault shape — primary use case)
- 10 wiki-style pages (longer, more entities per document)
- 5 chat log fragments (informal, abbreviations)
- 5 edge cases (ambiguous entities, contradictory facts, empty-extraction documents)

**Labeling:**
- Initial: developer annotates manually (50 docs = ~4 hours)
- Growth: LLM-assisted annotation — extract, correct, save corrections as gold standard
- Regression: every production bug adds its triggering document to the corpus

**Location:** `src/test/resources/eval/gold-corpus.json`

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| JSON schema validation | LLM output fails to parse against extraction schema | Retry (max 3) → DLQ if all retries fail |
| Entity type validation | Extracted type not in Schema Registry | Reject entity, log to DLQ |
| Source span verification | Entity text not found at claimed offset in source | Zero confidence → review queue |
| Max entity count | > 50 entities extracted from single chunk | Flag as potential over-extraction → review queue |
| Extraction rate limit | Configurable max documents/minute per connector | Throttle connector poll cycle |
| Token budget | LLM response > max_tokens | Truncation risk — log warning, validate partial output |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Entity detection F1 | Run gold corpus eval weekly (CI scheduled job) | Alert if F1 drops > 5% from baseline; investigate model or prompt change |
| Confidence calibration ECE | Recompute after every 500 review queue decisions | Recalibrate thresholds if ECE > 0.15 |
| False merge rate | Sample 20 resolved entities per week from production | Alert if false merge rate > 5%; tighten resolution threshold |

## 7. Production Monitoring

**Tracing Tool:** Spring Boot Actuator + Micrometer (already in stack). No external tracing tool for MVP — Tessera is self-hosted on IONOS VPS.

**Key Metrics to Track:**
1. `tessera.extraction.calls.total` — total LLM extraction calls (counter)
2. `tessera.extraction.calls.failed` — failed extraction calls (JSON parse failures, API errors)
3. `tessera.extraction.entities.extracted` — entities extracted per poll cycle
4. `tessera.extraction.review_queue.size` — current review queue depth per tenant
5. `tessera.extraction.cost.tokens` — total tokens consumed (input + output)

**Alert Thresholds:**
- Extraction failure rate > 20% over 1 hour → circuit breaker trips (existing `WriteRateCircuitBreaker` pattern)
- Review queue depth > 100 per tenant → operator notification
- Token cost > $50/day → cost alert

**Smart Sampling Strategy:**
- Log full extraction input/output for first 100 extractions (bootstrap calibration dataset)
- After bootstrap: log only failed extractions, review queue entries, and 5% random sample
- All extraction metadata (model, tokens, latency) on every call via Micrometer

## Corrections to CONTEXT.md

**CONTEXT.md Decision 7 (Embedding Storage) requires update:** Anthropic does not offer embedding models. The embedding provider is now:
- **Default:** Ollama (`nomic-embed-text`, 768 dimensions) — self-hosted in Docker Compose
- **Configurable:** OpenAI `text-embedding-3-small` (1536 dimensions) — per-tenant option
- **Abstraction:** Spring AI `EmbeddingModel` interface — provider-agnostic
- **Vector dimension:** Configurable per embedding model, not fixed. `entity_embeddings` table uses `vector(768)` for Ollama default; tenants using OpenAI get a separate dimension column or a wider vector.
