# Phase 2.5 — Context

**Goal (from ROADMAP.md):** Extend the connector framework with a second mode — LLM-based extraction of typed entities and relationships from unstructured text — so that free-text sources (wikis, notes, chat logs, emails, code commentary) land as first-class graph data that flows through the same rule engine, SHACL validation, reconciliation, and source authority matrix as structured connectors. Extracted candidates must be indistinguishable from REST-polled candidates downstream of the connector boundary.

**Depends on:** Phase 2 (connector framework, `ConnectorScheduler`, `ConnectorRunner`, `GraphService.apply()`, rule engine, SHACL, authority matrix, DLQ, Vault integration — all complete).

**Boundary:** 8 EXTR requirements. No MCP (Phase 3), no SQL/Kafka projections (Phase 4). The write funnel, rule engine, and SHACL remain untouched.

## Locked Decisions

These are answered. Downstream agents (researcher, planner) MUST treat them as constraints, not gray areas.

### 1. LLM Provider — Anthropic Claude via Spring AI

**Decision:** The unstructured extractor uses **Anthropic Claude** as the LLM provider, wired through Spring AI 1.0.x structured output support. The API key lives in Vault at `secret/tessera/llm/anthropic_api_key`.

**Why:** Matches the project's Claude-native LLM angle. Spring AI 1.0.x supports Claude structured output. Single Vault secret family (alongside connector bearer tokens). No second provider to configure, document, or rotate.

**How it shapes planning:**
- Spring AI dependency: `spring-ai-anthropic-spring-boot-starter` (1.0.x line, same BOM as Phase 3's MCP work).
- Vault bootstrap at startup: `ExtractionConfig` loads `anthropic_api_key` from Vault via Spring Cloud Vault Config Data API (same pattern as connector credentials).
- `ConnectorType = "unstructured-text"` carries a `provider` field in its `mapping_def` JSONB — defaulting to `"anthropic"`. The field exists for future extensibility but only `"anthropic"` is accepted in Phase 2.5.
- `ExtractionService` wraps the Spring AI `ChatClient` with a structured output call whose schema is built from the Schema Registry at call time (Decision 2).

### 2. Extraction JSON Schema — Live from Schema Registry (Caffeine-cached)

**Decision:** Each extraction call derives the output JSON schema from the **current Schema Registry state** for the tenant, using the Caffeine-cached `SchemaRegistry.loadFor(model_id)`. The schema is not baked into the connector config.

**Why:** Satisfies EXTR-03's "no redeploy" requirement — a new node type in the registry is extractable on the next document without any connector reconfiguration. The Caffeine cache (already populated for REST projection performance) makes this near-zero overhead.

**How it shapes planning:**
- `SchemaRegistrySchemaBuilder` utility class: walks the tenant's node types and edge types, builds a structured output schema (`ExtractionCandidate` class hierarchy or JSON schema string) that Spring AI sends to Claude as the expected response format.
- Schema includes: `node_type_slug`, `properties` (as per the type's declared properties), `relationships` (edge types and their endpoint type slugs).
- Adding a new type in the Schema Registry requires zero action from the connector operator — the next extraction call picks it up automatically.
- Cache invalidation: same TTL-based Caffeine cache as Phase 1/2. No special invalidation needed for the extraction path.

### 3. Connector Trigger Model — Scheduled Folder Scan (Existing SPI, No Changes)

**Decision:** The `UnstructuredTextConnector` implements the **existing `Connector.poll()` SPI** unchanged. It scans the configured folder on a fixed `poll_interval_seconds` schedule. Changed files are detected via file mtime + SHA-256 hash (`_source_hash`), stored in `ConnectorState`. Unchanged files are skipped before any extraction.

**Why:** Zero changes to `ConnectorScheduler`, `ConnectorRunner`, or the `Connector` SPI. The REST polling connector already proves the schedule-plus-hash pattern works. WatchService (file-event) would require a second dispatch path and adds OS-specific watch descriptor limits.

**How it shapes planning:**
- `UnstructuredTextConnector` (`type = "unstructured-text"`) scans the folder path from `mapping_def`. For each `.md` file: compute SHA-256 of content, compare against `ConnectorState`'s stored hash map, skip if unchanged.
- Changed or new files produce a `PollResult` with one or more `CandidateMutation` records (one per extracted entity).
- `ConnectorState` JSONB grows a `file_hashes: { "relative/path.md": "sha256hex" }` map. No schema change to the `connector_sync_status` table — JSONB.
- `mapping_def` for an unstructured connector carries: `folder_path`, `glob_pattern` (default `**/*.md`), `chunk_strategy` (default `paragraph`), `chunk_overlap_chars` (default `200`), `confidence_threshold` (default `0.7`), `provider` (default `"anthropic"`).

### 4. Text Chunking — Paragraph-based Default, Token-aware Option

**Decision:** Default chunking strategy is **paragraph-based** (split on blank lines, each paragraph = one chunk). Configurable alternatives: `sentence` and `token` (token-count-based with overlap). The default keeps Phase 2.5 free of a tokenizer dependency while covering the Obsidian vault use-case well. Chunk boundaries are deterministic for a given input + config.

**Why:** Obsidian markdown notes are paragraph-structured. Paragraph chunking is zero-dependency and reproducible. Token-based chunking would require a tokenizer (tiktoken or HuggingFace) — a separate dependency that can be added in a later phase when real documents exceed Claude's context window.

**How it shapes planning:**
- `TextChunker` strategy interface with `ParagraphChunker` (default), `SentenceChunker`, `TokenChunker` (Phase 2.5+ — can stub with `UnsupportedOperationException` and a clear error message).
- `chunk_strategy: "paragraph"` in `mapping_def` JSONB. Enum-validated on connector create (same closed-registry pattern as `Transform` enum in Phase 2).
- `chunk_overlap_chars: 200` — the last N characters of a chunk are prepended to the next chunk for context continuity. Applies to all strategies.
- Each chunk carries: `source_document_id` (SHA-256 of file path + mtime), `source_chunk_range` (char offset + length within the original document). These flow into provenance on the event log (EXTR-04).

### 5. Entity Resolution — Three-tier, Configurable Threshold

**Decision:** Entity Resolution Service (in `fabric-rules`) applies three tiers in order:
1. **Name + type exact match** — deterministic, no threshold.
2. **pgvector embedding similarity** — cosine similarity ≥ `resolution_threshold` (default `0.85`, per-tenant configurable in the Schema Registry or connector config). Match → merge into existing node.
3. **Fuzzy name match** — Levenshtein distance, invoked only if tier 2 yields no match above threshold. Rule-engine-pluggable (same `Rule` SPI as authority matrix rules).

Below the threshold after all three tiers → routed to `extraction_review_queue`.

**Why:** Satisfies EXTR-05's determinism requirement — for a given `(name, type, embedding)` against a fixed DB state, the resolution output is reproducible. The threshold is the single operator-tunable knob.

**How it shapes planning:**
- `EntityResolutionService` bean in `fabric-rules`. Called by `ConnectorRunner` BEFORE `GraphService.apply()` — it sets `targetNodeUuid` on the `CandidateMutation` if a match is found above threshold (merge path), or leaves it `null` (create path), or routes to the queue (below threshold).
- `resolution_threshold` in the unstructured connector's `mapping_def` JSONB (overrides tenant default).
- A test proves: same `(name, type, embedding)` + same DB state → same resolution decision on two back-to-back calls (EXTR-05 determinism requirement).
- `fabric-rules` ArchUnit test: `EntityResolutionService` must not call `GraphService` directly — only `ConnectorRunner` may do that.

### 6. Review Queue — Accept / Reject / Override UUID, Single-row API

**Decision:** The `extraction_review_queue` table stores candidates below the confidence threshold. Operator API:
- `GET /admin/extraction/review` — list with filters (`tenant`, `connector_id`, `type_slug`, `since`).
- `POST /admin/extraction/review/{id}/accept` — merge at current extracted properties; flows through `GraphService.apply()` with `extraction_confidence = 1.0` (operator-affirmed).
- `POST /admin/extraction/review/{id}/reject` — record reason, mark as discarded. Future polls of the same document will skip this entity if the hash is unchanged.
- `POST /admin/extraction/review/{id}/override` — body `{ "target_node_uuid": "..." }`. Operator supplies the correct existing node UUID. Flows through `GraphService.apply()` as an UPDATE on the specified node.

**Why:** No free-edit of extracted properties — editing raw JSON bypasses SHACL and creates unpredictable graph state. Operator can pick the correct merge target (override) but cannot rewrite what the LLM extracted. Accept/reject/override covers all practical operator decisions. Batch ops deferred — single-row is sufficient for MVP volume.

**How it shapes planning:**
- Flyway migration (next available after Phase 2's V11): `extraction_review_queue(id, model_id, connector_id, source_document_id, source_chunk_range, type_slug, extracted_properties JSONB, extraction_confidence NUMERIC, extractor_version TEXT, llm_model_id TEXT, resolution_tier TEXT, resolution_score NUMERIC, created_at, decided_at, decision TEXT, decision_reason TEXT, operator_target_node_uuid UUID)`.
- `ConnectorRunner` writes to `extraction_review_queue` when `EntityResolutionService` returns below-threshold outcome. This write is in a **separate transaction** from any graph mutation (there is no graph mutation for a queued candidate — it's pending).
- `ExtractionReviewController` under `/admin/extraction/review` (new controller, same module as `ConnectorAdminController`). RFC 7807 error shapes (Decision 8 from Phase 2).
- Accepted/overridden candidates call `GraphService.apply()` with the extracted properties and a `SourceType.UNSTRUCTURED` (new enum value alongside `STRUCTURED`).

### 7. Embedding Storage — Single Shared `entity_embeddings` Table

**Decision:** Embeddings are stored in a **single cross-type table**: `entity_embeddings(node_uuid UUID, model_id TEXT, embedding vector(N), created_at, updated_at)`. The vector dimension `N` is configurable per embedding model (default for Anthropic embeddings — confirm exact dimension during research). One Flyway migration installs pgvector and creates this table.

**Why:** Per-type tables would require dynamic Flyway migrations triggered by Schema Registry changes — a significant complexity spike. The single table works for all entity types, is searchable with a HNSW index, and requires one migration regardless of how many entity types declare embeddings.

**How it shapes planning:**
- Flyway migration (after AGE baseline, before extraction code): `CREATE EXTENSION IF NOT EXISTS vector;` + `CREATE TABLE entity_embeddings (...)` + `CREATE INDEX ON entity_embeddings USING hnsw (embedding vector_cosine_ops);`.
- `embedding` column declared optional per entity type via Schema Registry (`embedding_enabled: boolean`, `embedding_model: text`, defaulting to `null` / disabled).
- `EmbeddingService` bean (in `fabric-rules`): calls Spring AI's Anthropic embedding client, stores result to `entity_embeddings` after a node is created or updated. Called by `ConnectorRunner` post-apply.
- `EntityResolutionService` queries `entity_embeddings` for cosine similarity (pgvector `<=>` operator) when tier 2 runs.
- Vault path for Anthropic embeddings API key: `secret/tessera/llm/anthropic_api_key` (same key as extraction — Anthropic API key is shared across embedding and chat).

### 8. Provenance Schema — All Five Fields on Every Extracted Candidate

**Decision:** Every `CandidateMutation` produced by the unstructured path carries provenance metadata that flows into the event log and survives replay (EXTR-04):
- `source_document_id` — SHA-256 of `{connector_id}:{file_path}:{mtime_epoch}`.
- `source_chunk_range` — `{char_offset}:{char_length}` string (e.g., `"1024:512"`).
- `extractor_version` — version string of the `ExtractionService` (from Maven project version, injected via `@Value("${tessera.version}")`).
- `llm_model_id` — the Claude model string returned by the Spring AI response metadata (e.g., `claude-3-5-sonnet-20241022`).
- `extraction_confidence` — `BigDecimal` 0.0–1.0 from the LLM's structured output field.

**Why:** EXTR-04 is non-negotiable for replay. All five fields must be queryable per node. They live on the event log as part of `GraphMutation` metadata — not just in the review queue.

**How it shapes planning:**
- `GraphMutation` record grows optional provenance fields: `sourceDocumentId`, `sourceChunkRange`, `extractorVersion`, `llmModelId`, `extractionConfidence`. These fields are `null` for structured-connector mutations (no behavior change for Phase 2 code paths).
- `GraphService.apply()` writes provenance fields to the event log row when present.
- An ArchUnit test verifies that `CandidateMutation` produced by the unstructured path always has all five provenance fields non-null before reaching `ConnectorRunner`.
- `SourceType.UNSTRUCTURED` added to the existing `SourceType` enum (alongside `STRUCTURED`). Authority matrix already handles unknown source types via the conflict register.

### 9. First Concrete Connector — Markdown Folder / Obsidian Vault Shape

**Decision:** The reference implementation (`MarkdownFolderConnector`, `type = "unstructured-text"`) targets a local folder of Markdown files (`.md`). Configuration via `mapping_def`:
```json
{
  "folder_path": "/data/vault",
  "glob_pattern": "**/*.md",
  "chunk_strategy": "paragraph",
  "chunk_overlap_chars": 200,
  "confidence_threshold": 0.7,
  "provider": "anthropic"
}
```
The connector is green end-to-end in Testcontainers (AGE + pgvector in the same `apache/age:PG16_latest` container — pgvector is pre-installed in that image).

**Why:** Obsidian vaults are a natural fit for the Tessera use-case (notes → graph). Markdown is structurally simple. The same connector doubles as a test harness for the entire extraction pipeline.

**How it shapes planning:**
- Testcontainers: confirm `apache/age:PG16_latest` includes the `vector` extension; if not, build a custom Dockerfile extending it (same pattern as Phase 0's custom image).
- End-to-end IT (`MarkdownFolderConnectorIT`): seed a temp folder with `.md` files containing fictional entities, run one poll cycle, assert extracted nodes in AGE, assert provenance on event log, assert embeddings in `entity_embeddings`, assert high-confidence candidates merged, assert low-confidence candidates in review queue.
- WireMock not needed — extraction calls go to real Anthropic API or a Spring AI test double (Spring AI's `MockChatClient` where available).

## Success Criteria Recap (from ROADMAP.md)

The planner must produce plans that, when executed, satisfy:

1. An `UnstructuredTextConnector` ingests a text document → chunks → LLM extraction → `GraphService.apply()` — ArchUnit proves the path cannot bypass the write funnel.
2. Every extracted node carries all five provenance fields on the event log, queryable per node, surviving replay.
3. Entity resolution is deterministic: same `(name, type, embedding)` + same DB state → same merge decision twice.
4. Reconciliation between extracted and structured-source entities honors the authority matrix; conflicts logged to `reconciliation_conflicts` with extraction metadata intact.
5. pgvector installed via Flyway; embeddings optional per type; `MarkdownFolderConnector` green end-to-end in Testcontainers (AGE + pgvector).
