Create `docs/ops/tde-deployment-runbook.md` per D-09 and D-10. The runbook must cover all of the following sections with actionable commands for a Debian/Ubuntu-based IONOS VPS:

**Section 1: Overview**
- Purpose: Encrypt Postgres 16 data directory at rest using LUKS/dm-crypt on IONOS VPS
- Scope: Single-server self-hosted deployment (Docker Compose with `postgres-age`, `tessera`, `vault`)
- Prerequisites: Root/sudo access, IONOS VPS with additional block device or LVM partition for Postgres data, `cryptsetup` package installed

**Section 2: LUKS Volume Setup**
- Install cryptsetup: `apt-get install cryptsetup`
- Create LUKS container on block device: `cryptsetup luksFormat --type luks2 --cipher aes-xts-plain64 --key-size 512 --hash sha512 /dev/sdX`
- Open LUKS volume: `cryptsetup luksOpen /dev/sdX pg_encrypted`
- Create filesystem: `mkfs.ext4 /dev/mapper/pg_encrypted`
- Mount: `mount /dev/mapper/pg_encrypted /var/lib/postgresql/data`
- Set ownership: `chown -R 999:999 /var/lib/postgresql/data` (Postgres container UID)

**Section 3: Postgres Data Directory Configuration**
- Docker Compose volume mount: map `/var/lib/postgresql/data` from the encrypted mount into the `postgres-age` container
- Ensure Postgres `PGDATA` points to the encrypted volume
- Verify encryption is active: `cryptsetup status pg_encrypted` should show `cipher: aes-xts-plain64`

**Section 4: Auto-Unlock on Boot (with Key File)**
- Generate key file: `dd if=/dev/urandom of=/root/.pg_luks_key bs=512 count=1`
- Add key to LUKS slot: `cryptsetup luksAddKey /dev/sdX /root/.pg_luks_key`
- Set permissions: `chmod 0400 /root/.pg_luks_key`
- Add to `/etc/crypttab`: `pg_encrypted /dev/sdX /root/.pg_luks_key luks`
- Add to `/etc/fstab`: `/dev/mapper/pg_encrypted /var/lib/postgresql/data ext4 defaults 0 2`
- **Security note:** Key file on root partition means root compromise exposes the key. For higher security, use a remote key server or manual unlock. Document both options.

**Section 5: Key Rotation Procedure**
- Add new key to LUKS slot: `cryptsetup luksAddKey /dev/sdX` (enter new passphrase)
- Remove old key: `cryptsetup luksRemoveKey /dev/sdX` (enter old passphrase)
- Update key file if using auto-unlock
- Schedule: recommend quarterly key rotation
- **Note:** LUKS key rotation does NOT re-encrypt data. The master key stays the same. For full re-encryption: `cryptsetup reencrypt /dev/sdX` (requires LUKS2, Postgres must be stopped)

**Section 6: CMK-Encrypted Backups**
- Use `pg_dump` as in DR-DRILL.md, pipe through encryption: `pg_dump -Fc tessera | gpg --symmetric --cipher-algo AES256 --output /backup/tessera_$(date +%Y%m%d).dump.gpg`
- Store CMK (GPG passphrase) in HashiCorp Vault: `vault kv put secret/tessera/backup-cmk key="<passphrase>"`
- Backup rotation: keep 7 daily, 4 weekly, 3 monthly
- Verify backup integrity: `gpg --decrypt /backup/tessera_YYYYMMDD.dump.gpg | pg_restore --list | head`

**Section 7: DR Restore from Encrypted Backup**
- Provision new IONOS VPS
- Set up LUKS volume (repeat Section 2)
- Retrieve CMK from Vault: `vault kv get -field=key secret/tessera/backup-cmk`
- Decrypt and restore: `gpg --decrypt --passphrase "$CMK" /backup/tessera_YYYYMMDD.dump.gpg | pg_restore -d tessera`
- Verify graph is queryable: run smoke test from DR-DRILL.md
- Start Docker Compose stack and verify health

**Section 8: Monitoring**
- LUKS status check: `cryptsetup status pg_encrypted` (should show `active`)
- Disk health: `smartctl -a /dev/sdX` (S.M.A.R.T. monitoring)
- Filesystem space: `df -h /var/lib/postgresql/data`
- Add to cron (daily): script that checks LUKS status and disk health, alerts via email/webhook on failure
- Prometheus node_exporter metrics: `node_filesystem_avail_bytes`, `node_disk_io_time_seconds_total`

**Section 9: Troubleshooting**
- "LUKS volume won't open": check key file permissions, verify correct device path
- "Postgres won't start after reboot": check crypttab/fstab, ensure LUKS volume opens before Postgres container starts
- "Backup decryption fails": verify correct CMK, check GPG version compatibility
- "Performance degradation": AES-NI is hardware-accelerated on modern CPUs; verify with `grep aes /proc/cpuinfo`

Format the runbook with clear headings, numbered steps within each section, and shell commands in fenced code blocks. Include warning callouts for destructive operations.
