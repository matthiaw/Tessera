<objective>
Implement the hash-chained audit log for compliance-sensitive tenants (GxP, SOX, BSI C5). Each event row in `graph_events` records SHA-256(prev_hash || payload), forming an append-only tamper-evident chain. An on-demand verification job detects any broken link.

Purpose: Compliance frameworks require proof that audit records have not been tampered with after the fact. The hash chain provides cryptographic evidence of integrity without requiring external blockchain infrastructure.

Output: HashChain helper, modified EventLog.append() with inline hash computation, AuditVerificationService, AuditVerificationController, and concurrency + integrity tests.
</objective>

**HashChain.java** (package-private or public in `dev.tessera.core.events`, NO Spring annotations — pure function):
- `public static final String GENESIS_INPUT = "TESSERA_GENESIS";`
- `public static String genesis()` — returns `sha256(GENESIS_INPUT)`, 64-char lowercase hex
- `public static String compute(String prevHash, String payloadJson)` — returns `sha256(prevHash + payloadJson)`. Use `java.security.MessageDigest.getInstance("SHA-256")` and `java.util.HexFormat.of().formatHex(digest)` (JDK 17+). Throw IllegalArgumentException if prevHash is null.
- Private `static String sha256(String input)` using `StandardCharsets.UTF_8`.

**EventLog.java modification** (per D-C2):
1. Add `prev_hash` to the INSERT constant — extend the column list and VALUES list with `:prev_hash`
2. In `append()`, after computing sequence_nr but before the INSERT:
   - Query `model_config` to check if hash chain is enabled: `SELECT hash_chain_enabled FROM model_config WHERE model_id = :mid::uuid`. Cache this result per model_id in a `ConcurrentHashMap<UUID, Boolean>` to avoid per-event DB round-trip. Provide a `public void invalidateHashChainConfig(UUID modelId)` method for cache busting.

     a. Read predecessor hash: `SELECT prev_hash FROM graph_events WHERE model_id = :mid::uuid ORDER BY sequence_nr DESC LIMIT 1 FOR UPDATE` (strict FOR UPDATE, NOT SKIP LOCKED — Pitfall 5)
     b. If no predecessor exists (first event), use `HashChain.genesis()` as prevHash
     c. Compute: `String newHash = HashChain.compute(prevHash, payloadJson)` where payloadJson is the JSON-serialized payload

CRITICAL: The FOR UPDATE query must NOT use SKIP LOCKED. This is structurally different from OutboxPoller's SKIP LOCKED pattern. The hash chain requires strict serialization — concurrent appends to the same tenant MUST block until the predecessor commits.

CRITICAL: `graph_events` is partitioned by `event_time`. The V25 index `idx_graph_events_model_seq_desc ON (model_id, sequence_nr DESC)` enables efficient predecessor lookup. The `FOR UPDATE` on a partitioned table in PG16 is safe when the query uses an index — verify in the concurrency IT.

**AuditVerificationService** (@Component in `dev.tessera.core.audit`):
- Constructor: inject `NamedParameterJdbcTemplate`
- `public AuditVerificationResult verify(TenantContext ctx)`:
  1. Read all events for the tenant ordered by sequence_nr: `SELECT sequence_nr, prev_hash, payload FROM graph_events WHERE model_id = :mid::uuid ORDER BY sequence_nr ASC`
  2. Walk sequentially: for each event, compute expected hash = `HashChain.compute(previousEventPrevHash, payloadJson)` and compare with event's prev_hash
  3. For the first event, expected = `HashChain.compute(HashChain.genesis(), payload)`
  4. If mismatch found, return `AuditVerificationResult.broken(sequenceNr, expectedHash, actualHash, eventsChecked)`
  5. If all match, return `AuditVerificationResult.valid(eventsChecked)`
  6. Use cursor/streaming for large event sets — `jdbc.query()` with RowCallbackHandler, not `queryForList()` which loads all into memory
- CRITICAL: This runs OUTSIDE any write transaction — it is a read-only operation. Do NOT annotate with `@Transactional` or use `FOR UPDATE`. Use `@Transactional(readOnly = true)` for connection routing.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| EventLog.append() → graph_events | Hash chain computation inline in write transaction; prev_hash integrity is load-bearing |
| Admin API → AuditVerificationService | Verification endpoint is read-only but exposes hash values; must verify tenant ownership |
| Concurrent writers → FOR UPDATE lock | Race condition in hash chain could break integrity if lock is missing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-T2 | Tampering | graph_events.prev_hash | mitigate | Hash chain detects post-hoc tampering via sequential verification. `FOR UPDATE` in append() prevents concurrent chain corruption. AuditVerificationService walks the full chain to detect breaks. |
| T-04-R1 | Repudiation | EventLog hash chain | mitigate | SHA-256(prev_hash \|\| payload) creates a cryptographic audit trail. Each event is chained to its predecessor — modifying any event breaks the chain from that point forward. Compliance frameworks (GxP, SOX) require this for audit integrity. |
| T-04-S2 | Spoofing | AuditVerificationController | mitigate | JWT tenant claim check prevents cross-tenant verification requests. Same pattern as McpAuditController. |
| T-04-I2 | Information Disclosure | AuditVerificationResult | accept | Verification result includes hash values (expected vs actual) — these are SHA-256 digests, not reversible to payload content. Acceptable for admin-only endpoint. |
| T-04-D1 | Denial of Service | AuditVerificationService | mitigate | Verification walks all events sequentially — for tenants with millions of events this is slow. Mitigate via streaming (RowCallbackHandler), read-only transaction, and connection timeout. Future: batch verification with pagination. |
</threat_model>
